★ Flagship Project
JIMMY, Self-Healing AIOps Agent
Autonomous Kubernetes Incident Remediation  ·  AWS Bedrock + Claude AI
Project Overview
  • JIMMY is a production-grade autonomous AIOps agent running continuously on AWS EKS, monitoring a 7-microservice Kubernetes cluster and handling every infrastructure incident from detection to resolution without any human involvement.
  • The agent operates on a defined 7-step remediation workflow: ASSESS the cluster state, DIAGNOSE the failing component, CLASSIFY the incident type, fetch the appropriate RUNBOOK from S3, REMEDIATE the issue using Lambda action groups, VERIFY the fix, and REPORT the outcome via SNS email notification.
  • Every five minutes, AWS EventBridge triggers a cluster scan. Incidents handled autonomously include CrashLoopBackOff, OOMKilled, ImagePullBackOff, Readiness Probe Failure, High Restart Count, Pods Pending Too Long, and Service with No Endpoints.
Infrastructure and Architecture
  • The entire platform was provisioned with a single terraform apply command covering EKS cluster, VPC, ECR registries, Argo CD installation, Prometheus stack, and Grafana dashboards. No manual console work at any stage of setup.
  • GitHub Actions matrix CI was configured to spin 7 parallel build runners simultaneously, each pushing a separate Docker image to AWS ECR in parallel, cutting total pipeline time by 66 percent compared to sequential builds.
  • Argo CD Image Updater watches ECR for new image tags and automatically writes updated tags back to the GitHub repository, closing the full GitOps loop. Cluster state always reflects what is declared in Git.
  • Three-pillar observability stack: Prometheus scraping metrics at 15-second intervals, Grafana dashboards deployed via Kubernetes ConfigMap, and Fluent Bit DaemonSet forwarding all pod logs to Amazon CloudWatch for infrastructure-level root cause analysis.
  • Eight Lambda action groups were built using OpenAPI 3.0 specifications, covering log retrieval, metrics queries, cluster health checks, runbook fetching from S3, pod restart operations, and deployment scaling.
AI and AIOps Integration
  • Claude Haiku on AWS Bedrock serves as the reasoning engine for JIMMY. When an incident is detected, the agent sends structured context to Bedrock and Claude reasons across the operational scenario to determine the correct remediation action, even for edge cases not explicitly covered in the runbooks.
  • A Streamlit-based UI was built on top of the Bedrock integration, allowing engineers to query the cluster in natural language, ask operational questions, and investigate incidents through conversation rather than terminal commands.
  • Runbook-first enforcement ensures all remediation actions go through an S3-stored runbook before execution, providing consistent, auditable, and policy-compliant responses to every incident type.

OpenTelemetry Distributed Observability Platform
Production-grade  ·  20 Plus Microservices  ·  Three Deployment Tiers on AWS
Project Context
  • The OpenTelemetry Demo is a CNCF-maintained production-grade distributed application simulating a real e-commerce platform with over 20 microservices including a frontend, payment service, cart, checkout, recommendation engine, fraud detection, and Kafka event streaming.
  • Rather than running it once, this project deploys the same production-grade system three different ways on real AWS infrastructure, each representing a distinct level of infrastructure maturity. This demonstrates understanding of the full deployment evolution that most engineers never practice end to end.
Part 1, Docker Compose (Foundation)
  • Used the multi-stage Dockerfile and Docker Compose configuration to bring the entire 20-plus service stack live on an AWS EC2 instance with a single command.
  • Accessed the running application through the EC2 public IP on port 8080 and validated the full application was functional end to end, including the storefront, checkout flow, and service-to-service communication.
Part 2, AWS EKS (Cloud Native)
  • Provisioned a fully managed Kubernetes cluster on AWS using eksctl, moving beyond single-node Docker and into cloud-native multi-node orchestration.
  • Applied a single all-in-one Kubernetes manifest that created every Deployment, Service, and Pod in one operation, with all 19 pods reaching Running status across both worker nodes.
  • Used k9s for live cluster monitoring, tracking per-pod CPU and memory consumption, restart counts, and node distribution in real time, exactly as an SRE would observe a production workload.
Part 3, Argo CD GitOps (Production Maturity)
  • Installed Argo CD via Kubernetes manifests and connected it to the GitHub repository with automatic sync enabled, establishing a full pull-based continuous delivery pipeline.
  • Argo CD enforces that the cluster state always matches what is declared in Git. Any manual change applied directly to the cluster is detected as drift and automatically reconciled back to the desired state.
  • The final deployment showed 38 resources synced and healthy across 2 worker nodes, with the Argo CD application tree providing a full visual map of every deployed resource and its sync status.

ML + DevOps
YOLOv5 Object Detection on Kubernetes
ML Inference at Scale  ·  Production Kubernetes  ·  AWS EC2  ·  Zero-Downtime Operations
Project Overview
  • This project demonstrates the end-to-end deployment of a real-time machine learning inference application using production-grade DevOps practices on AWS infrastructure, bridging the gap between data science and platform engineering.
  • YOLOv5 is a state-of-the-art real-time object detection model. Making it production-ready required resolving deprecated OS-level dependencies, containerizing the application correctly, and building a deployment pipeline that treats every release as a first-class software artifact.
CI/CD and Containerization
  • Containerized the YOLOv5 application using a carefully crafted Dockerfile that resolved legacy OS dependency issues, producing a clean and reproducible container image ready for Kubernetes deployment.
  • Built a GitHub Actions CI/CD pipeline with a dual-tag image strategy, publishing both a latest tag and a commit SHA tag for every build. This ensures full deployment traceability, meaning any version ever deployed can be identified and rolled back to with a single command.
Kubernetes Infrastructure
  • Bootstrapped a multi-node Kubernetes cluster on AWS EC2 using kubeadm, with a dedicated control plane node and separate worker nodes, mirroring how production Kubernetes clusters are structured for real workloads.
  • Configured liveness and readiness probes along with resource limits for the inference deployment, enabling Kubernetes to automatically detect unhealthy pods and recover them without operator intervention.
  • Resolved a live disk pressure event on a running node by expanding the underlying EBS volume and extending the filesystem online, restoring pod scheduling capacity without any instance downtime or workload disruption. This was a real operational problem solved in a live environment.
